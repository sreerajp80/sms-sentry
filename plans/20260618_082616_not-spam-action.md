# Plan: "Not spam" action for Spam-section conversations

**Status:** completed

## What the issue is

In the Spam folder, selecting a conversation shows a contextual menu with only
**Mute notifications / Mark as read / Block sender / Delete**
([SmsOrganizerUi.kt:1014-1061](../app/src/main/java/in/sreerajp/sms_sentry/ui/SmsOrganizerUi.kt#L1014-L1061)).
There is **no way to move a conversation out of Spam**.

The capability exists only one-way:
- `reportSpam()` moves a message to Spam + flags it blocked
  ([SmsRepository.kt:177-180](../app/src/main/java/in/sreerajp/sms_sentry/data/SmsRepository.kt#L177-L180)),
  and the ViewModel also adds a `CONTACT â†’ Spam` rule
  ([SmsOrganizerViewModel.kt:351-368](../app/src/main/java/in/sreerajp/sms_sentry/ui/SmsOrganizerViewModel.kt#L351-L368)).
- The reverse data primitive `updateMessageCategory(id, category)` exists
  ([SmsRepository.kt:158-160](../app/src/main/java/in/sreerajp/sms_sentry/data/SmsRepository.kt#L158-L160))
  but is **not wired to any UI or ViewModel function**.

## Decisions (confirmed with user)

1. **Target folder = re-run the classifier** â€” each restored message is routed to its
   natural category (Finance / Reminder / Personal) instead of being dumped into Personal.
2. **Rules = remove + add allowlist** â€” delete any `CONTACT â†’ Spam` rule for the sender and
   add an allowlist entry so the sender is never auto-spammed again.

### Reconciling the two (the important nuance)

Re-running the classifier as-is on genuinely spammy text (e.g. *"You won!"*, *"90% DISCOUNT"*)
would just re-flag it back to Spam via `SPAM_KEYWORDS`
([SmsClassifier.kt:93-101](../app/src/main/java/in/sreerajp/sms_sentry/engine/SmsClassifier.kt#L93-L101)),
defeating the action. So the allowlist isn't a "force Personal" rule â€” it's a new
**`CONTACT â†’ NotSpam`** rule that the classifier treats as *"skip the spam heuristic for this
sender, but still classify Finance/Reminder/Personal normally."* This makes both the existing
messages and all future messages from the sender route correctly without ever auto-spamming.

## Notes / scope

- `"NotSpam"` is only ever a rule `targetCategory`, never a message category, so it won't
  appear as a folder or affect the folder tabs/filters.
- Existing `CONTACT â†’ Spam` rules are inserted with `OnConflictStrategy.REPLACE` on the
  autogen `id`, so they are **not** deduped by value â€” explicit delete-by-value is required
  (handled in `allowlistSender`).
- Optional (will include if you want): a Robolectric test asserting an allowlisted sender with
  spammy text classifies as Personal (not Spam), and that a bank-style body still â†’ Finance.

## Out of scope

- Changing the per-thread menu's existing "Report as spam" entry.
