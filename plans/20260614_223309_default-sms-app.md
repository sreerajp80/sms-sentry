# Plan: Make SMS Sentry a real default SMS app (two-way sync with the system SMS DB)

**Status:** completed

## Goal (from the user)

Let the user optionally set SMS Sentry as the **default SMS app**, behaving like Microsoft
SMS Organizer:

1. The app should contain **all SMS already on the phone** (import existing inbox/sent).
2. Deleting an SMS **in the app** must also delete it from the **phone's system SMS database**.
3. Going forward, incoming/outgoing SMS stay in sync between the app and the system provider.

## What this is today (baseline)

- The app keeps SMS only in its **own private Room DB** (`sms_organizer_database`); it never
  touches the system SMS provider (`content://sms`).
- It only listens to `SMS_RECEIVED` (passive) — see
  [SmsReceiver.kt](../app/src/main/java/in/sreerajp/sms_sentry/receiver/SmsReceiver.kt).
- It is **not** a default SMS app: no `SMS_DELIVER`/`WAP_PUSH_DELIVER` receiver, no
  `RESPOND_VIA_MESSAGE` service, no `sms:`/`smsto:` activity, no `RoleManager`/
  `ACTION_CHANGE_DEFAULT` flow, no `SmsManager` send, no runtime permission request.
- `sendSms()` only inserts into Room — it does **not** actually send.
- On empty start it seeds demo data ([SmsRepository.seedDemoData](../app/src/main/java/in/sreerajp/sms_sentry/data/SmsRepository.kt#L143)).
- `SMSMessage` has no link back to a system-provider row (no `_id`), so there is no way to
  delete the matching system message today.

## The core Android constraints (why this is non-trivial)

To even **appear** in Android's "Default SMS app" picker, an app MUST declare all four of:

1. A `BroadcastReceiver` for `android.provider.Telephony.SMS_DELIVER`
   (`android:permission="android.permission.BROADCAST_SMS"`).
2. A `BroadcastReceiver` for `android.provider.Telephony.WAP_PUSH_DELIVER` with mime type
   `application/vnd.wap.mms-message` (`android:permission="android.permission.BROADCAST_WAP_PUSH"`) — for MMS.
3. An `Activity` that handles `ACTION_SENDTO` for `sms:`/`smsto:`/`mms:`/`mmsto:` (compose).
4. A `Service` that handles `android.intent.action.RESPOND_VIA_MESSAGE`
   (`android:permission="android.permission.SEND_RESPOND_VIA_MESSAGE"`) — quick reply to calls.

Behavior changes once we are the default app:
- Incoming SMS arrives via **`SMS_DELIVER`** (not `SMS_RECEIVED`), and the default app is now
  **responsible for writing the message into the system provider** (the OS no longer does it).
- Only the default app may **write/delete** rows in `content://sms`.
- When NOT default, we can still **read** existing SMS (with `READ_SMS`) but cannot delete from
  the system DB; deletes then affect only our Room copy.

Play Store note (for the user's awareness, no code impact): apps using the SMS permission group
/ default-SMS role need a Play Console Permissions Declaration. Fine for sideload/personal use.

## Design decisions (recommended defaults — please confirm)

- **Dual mode.** The app keeps working when it is *not* default (passive listener + read-only
  import), and unlocks full read/write/delete + send when the user makes it default. This
  matches "if he wish."
- **MMS: fully handled (in scope).** Beyond the receiver/service needed for eligibility, the
  app will download incoming MMS from the carrier MMSC, parse the multipart PDU (text + media
  parts), persist them, and surface them in the inbox alongside SMS. Sending MMS is **not**
  included (send stays SMS-only); only receiving/parsing/storing/displaying MMS is in scope.
- **Real SMS sending** via `SmsManager` + writing to the Sent box, **only when the app is the
  default SMS app**. The current fake `sendSms` (which just inserts a Room row) is **removed**.
  When the app is **not** default, the send feature is **disabled** and the UI tells the user
  they must set SMS Sentry as the default SMS app to send messages (no fake send, ever).
- **No demo seeding when we have real-SMS access.** If `READ_SMS` is granted, import real
  messages instead of seeding demo data, to avoid polluting the real inbox / provider.
- **Provider mapping.** Add a nullable `systemId` (the provider row `_id`) to `SMSMessage` so a
  Room row can be mapped back to its system row for deletion/sync. Dedupe imports on `systemId`.
- **Delivery status (in scope).** Sent messages track a `status` (Sending → Sent → Delivered /
  Failed) via `sentIntent`/`deliveryIntent` `PendingIntent`s on `SmsManager.sendTextMessage`,
  surfaced in the UI. Note: delivery confirmation depends on the carrier — some networks never
  return a delivery report, so "Delivered" is best-effort, not guaranteed.
- **Conversation grouping (in scope).** The inbox is changed from a flat per-message list to a
  per-sender **conversation** list (one card per sender showing latest message + unread count),
  with a thread detail screen showing that sender's full history. Threading uses the provider
  `thread_id` when available, else the normalized sender address.

## Files to change / create

### New files
- `app/src/main/java/in/sreerajp/sms_sentry/data/SystemSmsStore.kt`
  Thin wrapper over `contentResolver` for `content://sms` (and `sms/inbox`, `sms/sent`):
  `readAll()` (returns rows with `_id`, address, body, date, type, read, thread_id),
  `writeInbox(...)`, `writeSent(...)`, `deleteById(systemId)`. All guarded so writes/deletes
  only run when we are the default app.
- `app/src/main/java/in/sreerajp/sms_sentry/util/DefaultSmsAppManager.kt`
  `isDefault(context)`, and `buildRequestIntent(context)` that uses `RoleManager.ROLE_SMS`
  (API 29+) or `Telephony.Sms.Intents.ACTION_CHANGE_DEFAULT` (API 24–28).
- `app/src/main/java/in/sreerajp/sms_sentry/receiver/SmsDeliverReceiver.kt`
  Handles `SMS_DELIVER`: reassemble PDUs, write to system provider Inbox, then funnel through
  `processAndInsertMessage` (carrying the new `systemId`), then notify.
- `app/src/main/java/in/sreerajp/sms_sentry/receiver/MmsDeliverReceiver.kt`
  Handles `WAP_PUSH_DELIVER` (mime `application/vnd.wap.mms-message`): receives the MMS
  notification (WAP push), then triggers a download of the actual MMS from the carrier MMSC via
  `SmsManager.downloadMultimediaMessage()` (API 21+), routing the result to the MMS parser.
- `app/src/main/java/in/sreerajp/sms_sentry/engine/MmsParser.kt`
  Reads downloaded MMS from `content://mms` + `content://mms/part`: extracts sender, date, text
  body (concatenating text parts), and media parts (image/audio/etc.), persisting media to app
  storage (or keeping a `content://` part reference). Feeds the text body through
  `processAndInsertMessage` so MMS is classified like SMS.
- `app/src/main/java/in/sreerajp/sms_sentry/data/SystemMmsStore.kt`
  Wrapper over `content://mms` (and `mms/part`) for reading MMS rows/parts and deleting by id —
  so MMS also participates in import and two-way delete.
- `app/src/main/java/in/sreerajp/sms_sentry/receiver/HeadlessSmsSendService.kt`
  Minimal `Service` handling `RESPOND_VIA_MESSAGE` (required for eligibility; can send via
  `SmsManager` for quick replies).
- `app/src/main/java/in/sreerajp/sms_sentry/receiver/SmsSendStatusReceiver.kt`
  Receives the `sentIntent` and `deliveryIntent` `PendingIntent` callbacks from a send and
  updates the message `status` (Sent / Delivered / Failed) in Room (Phase 3).

## Suggested implementation phases (each independently testable)

- **Phase 1 — Read & import + eligibility.** Schema migration (`systemId`/`type`/MMS columns),
  runtime permissions, `SystemSmsStore.readAll()`, `importFromSystemProvider`,
  `DefaultSmsAppManager` + Settings UI to request default, and the 4 required manifest
  components (the MMS receiver + service start as minimal stubs so the app is selectable as
  default; real MMS handling lands in Phase 4). After this: app shows all existing phone SMS
  and can become default.
- **Phase 2 — Two-way delete + live incoming.** `SmsDeliverReceiver` writes incoming to the
  provider + Room; `deleteMessage` deletes from the provider. After this: delete-in-app removes
  from phone DB; new SMS captured as default.
- **Phase 3 — Sending + delivery status.** Remove the fake `sendSms`. Real `SmsManager` send +
  Sent-box write + composer wiring + `RESPOND_VIA_MESSAGE`, **gated on being default**; send UI
  is disabled with an explanatory message when not default. Attach `sentIntent`/`deliveryIntent`
  and add `SmsSendStatusReceiver` so each sent message shows Sending → Sent → Delivered / Failed
  (delivery is carrier-dependent / best-effort). After this: app fully replaces the stock
  messaging app for SMS. (Note: the "disable send + explain when not default" behavior can also
  ship in Phase 1 by removing the fake send and gating the button on default status.)
- **Phase 4 — MMS (receive/parse/store/display).** Replace the Phase-1 MMS stubs with the real
  `MmsDeliverReceiver` (download via `SmsManager.downloadMultimediaMessage()`), `MmsParser`, and
  `SystemMmsStore`. Import existing MMS from `content://mms` (Phase 1's import extended), include
  MMS in two-way delete, classify MMS text, and render MMS (text + attachment) in the inbox and
  detail screens. Sending MMS remains out of scope.
- **Phase 5 — Conversation grouping.** Replace the flat per-message inbox with a per-sender
  conversation list (one card per thread: latest preview + unread count) plus a chat-style
  thread detail screen, threading on `thread_id`/normalized sender. Largely independent of the
  default-SMS work and can be sequenced before or after Phases 2–4.

## Risks / gotchas

- `fallbackToDestructiveMigration()` is on; the new migration must be correct or existing Room
  data is wiped on upgrade.
- Becoming default and writing/deleting in the real SMS DB is **destructive to the user's real
  messages** — needs clear confirmation UI and careful testing (ideally on an emulator first).
- Demo seeding and real import must not both run, or the real inbox gets demo spam.
- `targetSdk = 36`, `minSdk = 24`: must support both the `RoleManager` (29+) and legacy
  (`ACTION_CHANGE_DEFAULT`) default-app request paths.
- The existing `SmsReceiver` and the new `SmsDeliverReceiver` must not double-insert when
  default (guard by checking default status).
- **MMS is the hardest part.** MMS download depends on the carrier MMSC/APN and mobile data,
  is awkward to test on emulators (often needs a real SIM/carrier), and PDU/part parsing is
  fiddly. Expect MMS (Phase 4) to carry most of the risk and test effort; large media parts
  also raise storage and memory considerations.

## Test plan

- Unit/Robolectric: migration 2→3; `SystemSmsStore` read/delete with a fake `ContentProvider`;
  dedupe on import; delete routes to provider only when default.
- Manual (emulator): set as default via Settings; import shows existing SMS; send/receive SMS;
  delete in app and confirm it's gone from the stock Messages app; unset default and confirm
  read-only fallback.
- MMS (Phase 4, ideally on a real device with a SIM): receive an MMS with an image, confirm it
  downloads, parses, classifies, and renders with the attachment; import existing MMS; delete an
  MMS in-app and confirm it's removed from the stock app.
